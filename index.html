One chat. Built for every Olami team.

Bina (Hebrew: understanding) is Souled's in-house AI assistant. Every Olami/Souled staff member — and now every partner-org contact — gets a chat window tailored to exactly what their role should see, and nothing they shouldn't.

Coaches get a sidekick for handbook lookups, their caseload, and student history. Regional directors get scoped student data. Fundraisers get aggregate counts and ready-to-share lists. Partner orgs get visibility into the students they referred. Exec gets full access. One chat. Six signed-in realities.

The big idea. Bina knows who's signed in. A coach asking about her own student gets a different reply than a fundraiser asking the same question — and a regional director gets something different again. The system checks every request against the user's role before any answer is generated, so sensitive details never leak across teams.

If a coach reports a student in real distress, Bina responds supportively and immediately emails the admin team.

👑 Exec · 🗺️ Regional · 📊 Analyst · 📈 Aggregate tools
🧠Ask any data question in plain English

No need to know any query language — just ask. Bina translates the question into a database query, checks it against the user's permissions, and returns the answer. Test records and blocked fields are filtered out before any results are shown.

🪄Honest escalation when she can't answer

If a question is qualitative or asks about something Souled doesn't track, Bina says so — no guessing. She flags the conversation, notifies the Souled Data Team, and tells the user a human will follow up.

🤝 Referring-Org tools
📜Students we referred

Browse the students your org sent to Souled — registration history, classes attended, trips joined, broad engagement signals (active / inactive).

📚Souled catalog

Search upcoming events, classes, and the public Souled video library — handy for sharing with prospective students.

💬 Personal conversation first, not a cold warning. You're the one with a relationship with her — gently let her know there are consequences.

What gets checked before Bina answers a data question

1. Only allowed data sources
Bina can only read from a defined list of Salesforce tables — students, coaches, registrations, etc. Anything else is rejected before it runs.
2. Sensitive fields stay hidden
Each user type has fields that are off-limits — phone numbers, pay, narrative notes, AI reasoning. The check catches them in the question's selection list before any data is fetched.
3. Counts vs. lists, by user type
Aggregate fundraisers can ask for counts and rollups, but never row-level lists. State and country can appear in counts but never alone.
4. Test records auto-excluded
Bina automatically filters out test contacts so reports never accidentally include them. Any attempt to modify or delete data is also blocked.

What makes Bina special

Smart caching

The instructions Bina reads on every turn are cached, so each follow-up question is much cheaper than the first. Cache-hit rate is visible in the Cost tab.

Always-fresh access checks

When a coach asks about a student, Bina re-checks her current roster every single time. So if a student moves to a different coach, the prior coach can't accidentally still query them.

Knowledge updates without downtime

When the handbook or library changes, an admin uploads the new content and Bina starts using it within seconds. No code release, no maintenance window.

Monitoring that can't break chat

Conversation tracking runs in a separate place from the chat itself. Even if monitoring breaks, chat keeps working — and there's a one-flag rollback if needed.

Cost figures that don't lie

Each conversation's cost is computed and saved at the time it happens. So when AI prices change next quarter, last month's reports still tell you what last month actually cost.

She says "I don't know" instead of guessing

Bina has a defined list of things Souled tracks. Ask about something undefined — like a metric we don't measure — and she says so plainly and flags it for the data team, instead of inventing an answer.
